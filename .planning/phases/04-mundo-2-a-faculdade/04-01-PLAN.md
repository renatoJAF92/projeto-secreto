<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SaveManager ↔ Player | active_power String must be valid power ID or empty; invalid IDs silently ignored |
| Project.godot ↔ Player | Input actions must exist or Input.is_action_just_pressed crashes |
| Player ↔ Projectile (later) | Player spawns projectiles; spawned projectiles must not overlap player collider at birth (handled in Plan 03) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | **I** (Information Disclosure) | SaveManager load_game() | mitigate | v2→v3 migration happens silently on first load; no user warning needed, save is transparent |
| T-04-02 | **T** (Tampering) | player._current_power string | accept | Powers are cosmetic mechanics (no economic value); invalid power IDs just fail silently (use_power() returns early) |
| T-04-03 | **A** (Availability) | Project.godot input actions | mitigate | Both actions hardcoded in project.godot at init; editor warns if not found; fallback: Input.is_action_just_pressed checks bool exists |
| T-04-04 | **D** (Denial) | active_power persistence | mitigate | SaveManager.save_game() called after unlock_power() and cycle_power(); file I/O errors logged by Godot engine, game continues with in-memory state |

**Rationale:** Phase 4 has no network, authentication, or encryption concerns. SaveManager schema bumps are internal only; players' old saves upgrade transparently. No PII or sensitive data involved.
</threat_model>
