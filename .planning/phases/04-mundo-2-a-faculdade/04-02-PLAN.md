<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Player ↔ Checkpoint | Player collider triggers checkpoint Area2D; coordinate trust (both in same scene) |
| Player HP ↔ SaveManager | HP is transient (reset on scene load), not persisted in save (saves only powers_unlocked, worlds_completed) |
| AudioManager ↔ Checkpoint | play_sfx() may receive key not in registry; handled gracefully with .exists() check |
| Power HUD ↔ Player | HUD reads player._current_power and _power_cooldown; no write-back (read-only display) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | **I** (Information Disclosure) | HP display HUD | accept | HP is visible to player on-screen (intentional), no hidden state |
| T-04-06 | **T** (Tampering) | Checkpoint _activated flag | mitigate | One-shot flag prevents infinite heal loops; collision disabled after activation ensures no double-trigger |
| T-04-07 | **A** (Availability) | AudioManager.play_sfx() missing key | mitigate | Register all Phase 4 keys upfront; ResourceLoader.exists() prevents crash on missing .wav; logs warning instead |
| T-04-08 | **D** (Denial) | Player death reset (respawn) | accept | Death is by design (1-hit → respawn in Mundo 1, multi-hit in Mundo 2+); no abuse vector |
| T-04-14 | **I** (Information Disclosure) | Power HUD state visible | accept | Power availability/cooldown is intentional gameplay feedback; displayed on-screen by design |

**Rationale:** HP, checkpoint, and power HUD systems are cosmetic game mechanics with no economic or security implications. Audio registration is defensive (graceful degradation on missing files). Power display is purely visual feedback (read-only from player state).
</threat_model>
